# 関連情報の整理（参考文献・上流・バックログ）

レポートを支える外部情報の網羅性と鮮度を点検する。対象は
参考文献章（`content/REPORT.md` 第 10 章相当）、本文中の出典タグ [n]、
各 `content/method-*.md` の参考リンク節、`docs/BACKLOG.md`、上流リポジトリへの言及。

## 手順

1. 参考文献章の一覧を取得し、本文中の出典タグ [n] と突き合わせる
   （`grep -o '\[\\?\[[0-9]+\]' content/*.md` 等で機械的に収集する）。
2. 各ページの参考リンク節を集約し、重複・矛盾（同じ対象に別 URL）を確認する。
3. `docs/BACKLOG.md` を読み、レポートの残課題章・engineering-notes・直近の作業内容と突き合わせる。
4. ネットワークが許せば、主要リンク（上流リポジトリ・論文・データ取得元）の生存確認を行う。
5. 軽微な問題（タグ番号ずれ、リンク表記ゆれ）は修正し、文献の追加・差し替えは提案に留める。

## チェックリスト

### 参考文献・出典タグ
- [ ] 本文中の全出典タグ [n] が参考文献章に存在するか（欠番・未使用文献はないか)
- [ ] 別ページから本体の参考文献を参照する場合（`index.html#refN` 等）、アンカーが実在するか
- [ ] 論文には arXiv 等の安定した識別子、リポジトリにはコミット/バージョン情報があるか
- [ ] 「公式に文献が存在しない」ことを述べる場合、その旨が確認日とともに明記されているか

### 上流リポジトリ情報
- [ ] 上流（このリポジトリでは google/dpsynth・mbi 等）の参照がバージョン/ハッシュ付きか
- [ ] 上流の更新で陳腐化しうる記述（「ドキュメントに記載がない」「バグがある」等）に
      確認時点の情報であることが示されているか
- [ ] 上流に還元すべき内容（パッチ・バグ報告）がバックログに反映されているか

### バックログ
- [ ] テーブル形式（# / タイトル / ラベル / 概要）が保たれ、`create_issues.sh` で機械処理できる形か
- [ ] 直近のセッション・コミットで対応した項目に ✅ と残作業が追記されているか
- [ ] レポートの残課題章・各レビューで見つかった新規課題が漏れなく載っているか
- [ ] Issue 化済みの項目がある場合、Issue 番号への参照があるか
